## 1. 쇼핑몰 바로가기

**🛍️ [https://laos-coffee.vercel.app](https://laos-coffee.vercel.app)** — 클릭하면 바로 쇼핑몰로 이동합니다.

## 2. 과제에 대한 답 — 내 서비스에 백엔드 붙이기 (갈래 B)

지난 시간의 PRD·목업을 이어, 리뷰 기능 하나에 실제 백엔드(Supabase)를 붙이고 배포까지 마친 결과입니다.

### 2-1. 새로고침해도 데이터가 남는 장면

상품 상세 페이지(예: [루앙프라방 유기농 아라비카 원두](https://laos-coffee.vercel.app/products/1))에서 로그인 후 리뷰를 남기면, DB(Supabase Postgres)에 저장되어 새로고침·재접속·다른 기기에서도 그대로 보입니다. 실제로 남긴 테스트 리뷰가 지금도 남아있는 것을 아래처럼 확인할 수 있습니다.

```
$ curl "https://.../rest/v1/reviews?product_id=eq.1&select=author_name,rating,created_at&order=created_at.desc&limit=1"
[{"author_name":"전대진2","rating":4,"created_at":"2026-07-16T12:43:09.234189+00:00"}]
```

로컬 개발 서버에서 작성한 리뷰가 배포된 실제 사이트([laos-coffee.vercel.app/products/1](https://laos-coffee.vercel.app/products/1))에도 그대로 보였습니다 — 브라우저나 기기가 아니라 서버(DB)에 저장된다는 뜻입니다.

### 2-2. 설계 한 장 (지도)

| 항목 | 내용 |
|---|---|
| 백엔드가 꼭 필요한 기능 | 리뷰 작성/조회 — 내가 쓴 리뷰가 새로고침해도, 다른 기기로 접속해도 그대로 남아야 함 |
| 화면만으로 되는 기능 | 상품 목록/상세(목업 데이터), 장바구니, 주문·결제(로컬 시뮬레이션) — 아직 실제 재고·결제 연동 전이라 서버 없이도 충분 |
| 필요한 창구(엔드포인트) | 리뷰 목록 조회(Read) · 리뷰 작성(Create). 삭제/수정은 이번엔 범위 밖 |
| 다뤄야 할 비밀값 | Supabase 프로젝트 URL + anon key → `.env.local`로 분리, GitHub·Vercel 양쪽 다 코드에 직접 안 넣고 환경변수로 등록 |
| 호스팅 | ③ Supabase — "저장·조회·로그인"이 전부라 직접 서버(Render)를 세울 이유가 없고, 무료 티어로 충분. RLS로 보안도 확보 가능 |

**스키마**: [`docs/dev-journey/supabase-reviews-schema.sql`](./docs/dev-journey/supabase-reviews-schema.sql)

### 2-3. 고른 호스팅과 이유

**Supabase(BaaS)를 선택했습니다.** 필요한 게 "저장·조회·로그인"뿐이라 직접 서버를 세울 이유가 없기 때문입니다 — ①집 컴퓨터+터널은 상시 서비스에 안 맞고, ②Render는 커스텀 서버 코드가 필요한데 우리가 하려는 건 순수 CRUD라 과합니다.

## 3. 제출 전 보안 체크

| 체크 항목 | 결과 | 근거 |
|---|---|---|
| 비밀값(키·비밀번호)을 코드에 직접 안 적고 환경변수로 뺐는가? | ✅ 예 | 소스코드 전체(`src/`)에서 Supabase URL·키 하드코딩 검색 → 0건. `.env.local`은 `git log`상 커밋 이력 없음(`.gitignore`의 `.env*`로 항상 제외). 실제 값은 로컬 `.env.local`과 Vercel 환경변수(Production/Preview/Development)에만 존재 |
| 가격·수량·권한 같은 검증을 백엔드에서 하는가? (프론트만 믿지 않기) | ⚠️ 부분적으로만 | 리뷰의 `rating`은 DB에 `CHECK (rating between 1 and 5)` 제약이 걸려있어 API를 직접 호출해도 범위 밖 값은 DB가 거부합니다. 다만 **장바구니·주문의 가격/수량은 백엔드 검증이 없습니다** — 설계상 이 부분은 서버가 없는 로컬 시뮬레이션이라 범위 밖이며, 실제 결제 연동 시 반드시 서버 검증이 필요합니다 |
| (Supabase를 쓴다면) RLS를 켜고 정책을 넣었는가? | ✅ 예 | 직접 재검증: 인증 없이 조회(SELECT) → 200 정상 응답(공개 정책), 인증 없이 작성(INSERT) 시도 → 401 `"new row violates row-level security policy"`(차단), 인증 없이 삭제(DELETE) 시도 → API는 204를 반환하지만 삭제 정책 자체가 없어 실제로는 0건 처리(재조회로 데이터 그대로 남아있음을 확인) |
| 로그인이 필요한 데이터에 권한 확인(인가)이 들어가는가? | ⚠️ 기능별로 다름 | 리뷰 작성은 ✅ — RLS의 `auth.uid() = user_id` 조건이 서버(DB) 단에서 로그인 토큰을 검증하는 실제 인가입니다. 반면 체크아웃·주문내역 페이지는 `if (!user)` 형태의 화면단 가드만 있고 데이터 자체가 서버에 없어(로컬 시뮬레이션) 백엔드 인가는 해당 없음 |
| 사용하는 무료 티어의 한도와 요금 알림을 아는가? | ✅ 예 | Supabase Free 플랜: DB 500MB, 파일 스토리지 1GB, 월 egress 5GB, MAU 5만명, 무료 프로젝트 최대 2개, **1주일 미사용 시 자동 일시정지**. 카드 등록 없이 가입 가능해 과금 위험은 낮음(대시보드 Billing에서 결제수단 등록 여부는 별도 확인 권장) |

## 프로젝트 개요

- **타깃 사용자**: 라오스·베트남·캄보디아 여행 경험이 있는 20~40대, 해외여행/현지커피/유기농 식품을 선호하는 사람
- **핵심 가치**: 여행의 추억을 다시 경험하게 하는 커피 쇼핑몰
- **개발 범위(현재)**: 대부분 프런트엔드 중심으로 진행하되, 리뷰·로그인은 Supabase(Postgres + Auth)로 실제 저장/인증됩니다. 결제 연동은 이후 단계에서 진행

## 실행 방법

```bash
npm install
npm run dev
```

`http://localhost:3000`에서 확인합니다.

## 배포

[Vercel](https://vercel.com)에 배포되어 있습니다. GitHub 저장소와 연동되어 있어 `main` 브랜치에 push하면 자동으로 재배포됩니다 (수동 배포가 필요하면 `vercel deploy --prod`).

## 기술 스택

- **Framework**: Next.js (React 기반)
  - SEO 노출에 유리 (라오스 커피, 유기농 커피, 여행 선물 등 키워드 검색 대응)
  - AI 코드 생성 안정성이 높고 한국어 학습 자료가 풍부
  - 로그인/회원가입/장바구니/주문/결제/리뷰/관리자 페이지 등 향후 기능 확장이 용이한 구조
- **Backend**: Supabase (Postgres + Auth) — 리뷰·로그인만 실제 연결, 나머지(상품·장바구니·주문)는 목업/로컬 상태 (자세한 내용은 위 "2. 과제에 대한 답" 참고)

## 핵심 화면 (MVP)

| 화면 | 목적 | 주요 표시 정보 |

| 장바구니 | 구매 예정 상품 관리 | 상품 목록, 수량 변경, 상품 금액, 배송비, 총 결제금액, 주문하기 버튼 |

## 목업 데이터

상품 목록/상세는 백엔드 없이 목업 데이터로 진행합니다. 상품 데이터 구조 예시:

```json
{
  "id": 1,
  "brand": "Lao Aroma",
  "name": "루앙프라방 유기농 아라비카 원두 500g",
  "description": "여행의 아침을 다시 떠올리게 하는 라오스 대표 유기농 원두",
  "price": 18500,
  "salePrice": 16800,
  "discountRate": 9,
  "category": "원두",
  "weight": "500g",
  "origin": "라오스 볼라벤 고원",
  "rating": 4.9,
  "reviewCount": 238,
  "likeCount": 412,
  "stock": 86,
  "delivery": "무료배송",
  "tags": ["BEST", "유기농", "여행추천"],
  "isNew": true,
  "isBest": true,
  "image": "루앙프라방 커피 농장을 배경으로 한 브라운 컬러 원두 패키지",
  "soldOut": false
}
```

실제 구현 시 동일한 구조의 상품 데이터를 최소 6개 이상 사용합니다. 상품 이미지는 실사 촬영 전이라, 상품 설명(원산지·풍경 모티프)에 맞춰 procedural하게 그린 SVG 일러스트로 대체했습니다 (`src/components/ProductIllustration.tsx`).
